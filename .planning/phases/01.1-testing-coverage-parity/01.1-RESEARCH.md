## 1. RTL with React 19 + Next 16 App Router (Vitest 4)

The Sandra CRM pattern is current. Confirmed against the Next 16 docs at `node_modules/next/dist/docs/01-app/02-guides/testing/vitest.md`: jsdom environment, `@testing-library/react@^16` for React 19, co-located `*.test.tsx` files. Two notes for the planner:

- `@testing-library/react@^16` (Dec 2024) added React 19 support and made `@testing-library/dom` a peer dependency. npm 11 auto-installs peer deps but it is safer to add `@testing-library/dom` explicitly to avoid version drift between unrelated installs. See https://github.com/testing-library/react-testing-library/releases.
- The Next docs example uses `@vitejs/plugin-react`. Sandra CRM does not, and Vitest 4 transforms TSX natively via esbuild. Match Sandra CRM (no plugin-react). Add it later only if a test fails because of a JSX transform issue.

Suggested deps to install (versions verified 2026-04-30 against the npm registry):

## 2. Server Components in jsdom

Next 16's Vitest guide states this directly: "Since async Server Components are new to the React ecosystem, Vitest currently does not support them. While you can still run unit tests for synchronous Server and Client Components, we recommend using E2E tests for async components." See `node_modules/next/dist/docs/01-app/02-guides/testing/vitest.md` lines 9 and onward.

Plan accordingly:

- RTL targets Client Components (forms, runners, navs, leaf interactive nodes). Anything with `"use client"` is fair game.
- Sync Server Components without async data fetches can render in jsdom but it is rarely worth it; the integration suite + e2e suite already cover server-side behaviour with higher fidelity.
- Async Server Components (the common case in this app — most `page.tsx` files do `await supabase...`) are Playwright territory only. Plans must not place them in `*.test.tsx`.

Source: Next.js App Router Vitest docs (in tree, Next 16.2.4).

## 3. Playwright 1.59 storage-state pattern

The Sandra CRM pattern (one-time auth via `e2e/auth.setup.ts`, write to `e2e/.auth/user.json`, every other project depends on `setup` and reuses `storageState`) is current and matches Playwright 1.59. BMH Institute's existing `playwright.config.ts` already wires this up identically. The only file missing is `e2e/auth.setup.ts` itself (and `e2e/fixtures.ts`).

What changed between Sandra CRM's setup and BMH Institute:

- BMH Institute `webServer.command` is `npx next dev -p 3200` (port 3200). Specs must use `baseURL` from the Playwright config or relative paths, never hardcode `localhost:3100`.
- The fixtures guard in BMH Institute must check the prod ref `dhvfsyteqsxagokoerrx` and refuse to run if the URL contains it. Easy copy-paste-miss from Sandra CRM's `copflsklaefwzipsrjqz` — the planner's verification step should grep for the wrong ref.

Multiple storage states (admin vs learner) are supported by adding two `setup` projects (one per role) and two browser projects, each with its own `storageState`. Phase 01.1 only needs one (a learner who happens to also be admin via `ADMIN_EMAILS`) — admin-vs-learner separation is a Phase 04 concern (TEST-01..03). Note this for the planner: do not over-engineer fixtures here.

Reference snippet for `e2e/auth.setup.ts` (adapted from Sandra CRM):

```ts
// Saves storage state at e2e/.auth/user.json. Every other spec inherits.
import { test as setup, expect } from "@playwright/test";
import { adminClient, ensureTestUser, TEST_USER_EMAIL, TEST_USER_PASSWORD } from "./fixtures";

const AUTH_FILE = "e2e/.auth/user.json";

setup("authenticate", async ({ page }) => {
  await ensureTestUser(adminClient());
  await page.goto("/login");
  await page.getByLabel(/email/i).fill(TEST_USER_EMAIL);
  await page.getByLabel(/password/i).fill(TEST_USER_PASSWORD);
  await page.getByRole("button", { name: /^sign in$/i }).click();
  await page.waitForURL((url) => !url.pathname.startsWith("/login"), { timeout: 15_000 });
  await expect(page.getByRole("button", { name: /sign out/i })).toBeVisible();
  await page.context().storageState({ path: AUTH_FILE });
});
```

## 4. Supabase ephemeral branches for Path B (write-path e2e)

Feasible but not free, and provisioning latency is undocumented. Sources cross-checked at https://supabase.com/docs/guides/deployment/branching, https://supabase.com/docs/guides/deployment/branching/working-with-branches, https://supabase.com/docs/guides/platform/manage-your-usage/branching, and https://supabase.com/docs/reference/api/v1-create-a-branch.

Findings:

- Branches are dataless by default. They start from a fresh schema apply; `seed.sql` runs once at branch creation if present. CI fixtures would need to seed every required row through the service-role client after the branch comes up.
- Programmatic create + delete is supported via the Management API (`POST /v1/projects/{ref}/branches`, `DELETE /v1/branches/{branch_id}`). The Supabase MCP exposes `create_branch`, `list_branches`, `delete_branch` which call the same endpoints.
- Cost: Branching Compute Hours are billed separately and are NOT covered by the standard $10/month Compute Credit. Even Micro compute accrues Branching Compute Hours per branch hour. A 30-hour branch will appear as a line item on the invoice. Spend cap does not protect this.
- Provision time is not in the public docs. Community reports range from 30 seconds to a few minutes. Treat as undocumented and validate with a one-off probe before assuming it fits inside a Playwright `globalSetup` timeout.
- Persistence: preview branches auto-pause on inactivity and auto-delete when their associated PR closes. For a CI run not tied to a PR, persistence is the caller's job — explicit `delete_branch` in `globalTeardown`. If the teardown errors, the branch lingers and bills.

Verdict for Phase 01.1: viable but adds three things the phase did not budget for — a branch-lifecycle helper, a seed step, and a billing line item. Path A (default in CONTEXT) avoids all three. Recommend planner stays on Path A and lists Path B as a follow-up phase only if Jarrad locks in to it.

If the planner does pursue Path B, the lifecycle skeleton is:

```ts
// e2e/global-setup.ts (Path B only — confirm with Jarrad before authoring)
import { createClient } from "@supabase/supabase-js";

export default async function globalSetup() {
  // 1. POST /v1/projects/{prod-ref}/branches with a unique branch name
  // 2. Poll list_branches until status === "ACTIVE_HEALTHY"
  // 3. Read branch credentials, write into process.env.TEST_SUPABASE_*
  // 4. Apply seed via service-role client
}
```

```ts
// e2e/global-teardown.ts (must run even on failure)
export default async function globalTeardown() {
  // DELETE /v1/branches/{branch-id} — log but do not throw if already gone
}
```

## 5. `@testing-library/user-event` v14+ async API

Default to v14 async API. Sandra CRM uses it everywhere (`userEvent.setup()` then `await user.click(...)` / `await user.type(...)`) and v14 has been the standard since 2022; v13 sync API is deprecated. Plans must call `setup()` once per test (or per render) and `await` every interaction, otherwise React state updates fire outside `act()` and tests flake. Reference: https://testing-library.com/docs/user-event/intro/.

```tsx
import userEvent from "@testing-library/user-event";

const user = userEvent.setup();
await user.click(screen.getByRole("button", { name: /sign in/i }));
await user.type(screen.getByLabelText(/email/i), "claude@test.com");
```

## 6. Mocking Server Actions in RTL

Server Actions exported from `actions.ts` look like normal async functions to the bundler in tests. The Sandra CRM pattern is `vi.mock("./actions", () => ({ ... }))` with each exported action replaced by `vi.fn()`. This is the recommended approach for Phase 01.1; calling real Server Actions from jsdom would attempt to hit Supabase and fail.

```tsx
// alongside <SomeForm /> in some-form.test.tsx
vi.mock("./actions", () => ({
  submitInvite: vi.fn(async () => ({ ok: true })),
}));
```

A few practical notes:

- Mock paths are relative to the test file, not the project root. `./actions` works because the form lives in the same directory.
- For actions that import from `@/lib/...`, the `@/lib/...` import is resolved by the alias in `vitest.rtl.config.ts` and only matters if the test reaches into that module. If the action is mocked, the lib never loads.
- `next/navigation` (`useRouter`, `useSearchParams`, `usePathname`) needs the same `vi.mock` treatment; jsdom does not ship with Next router context. Sandra CRM has a copy-paste mock block for this — the planner should propagate it to any test that renders a component using router hooks.

## Pitfalls for the planner

1. The CONTEXT says `e2e-prod/` is missing — it is not. Do not regenerate or overwrite the existing `auth.setup.ts`, `admin.spec.ts`, `dashboard.spec.ts` in `e2e-prod/`.
2. Copy-pasting Sandra CRM's `e2e/fixtures.ts` will leave `copflsklaefwzipsrjqz` (the Sandra dev project ref) in the guard. The planner must replace it with `dhvfsyteqsxagokoerrx` before the file lands, or the guard will not protect BMH Institute prod data.
3. Do not put async Server Components into `*.test.tsx`. Vitest cannot render them per the Next 16 docs. Plans for any test that touches a `page.tsx` or `layout.tsx` belong in integration (mocking Supabase) or e2e (real Next dev server), not RTL.
4. `vitest.rtl.config.ts` must `exclude` `**/*.integration.test.ts` and `e2e/**` or the RTL run will pull in Postgres-backed tests and Playwright specs, both of which fail in jsdom.
5. Path B (Supabase ephemeral branches) costs Branching Compute Hours that bypass the Compute Credit and the Spend Cap. Do not plan Path B without an explicit lock from Jarrad on the cost trade.
