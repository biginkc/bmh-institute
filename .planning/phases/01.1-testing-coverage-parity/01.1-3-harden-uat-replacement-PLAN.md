Storage-state choice for the new HARDEN-01 spec (decided per the user prompt's hard constraint #8 — see Open Questions for justification):

**The new spec uses fresh in-spec sign-in, NOT a second setup project.** Justification:
- Path A scope: minimal delta. A second setup project requires modifying `playwright.prod.config.ts` (adding a project, adding a learner storage-state path) and creating a second `auth.setup.ts`. The current spec just needs to sign in once at the top.
- Account discipline: the existing `E2E_TEST_EMAIL` is the admin/owner per `playwright.config.ts` `webServerEnv.ADMIN_EMAILS`. A learner-only account would need to be created on prod and provisioned with deliberately-empty role-group access. That account does not yet exist; provisioning it in Path A is out of scope.
- However the contract HARDEN-01 enforces is "any non-admin session is redirected away from /admin/reports/*". A clean session (no cookies, no storage state) IS a non-admin session, and is the strictest test of the guard — it exercises both the unauthenticated redirect (`/login`) AND the redirect-after-sign-in path. We can sign in fresh as the existing `E2E_TEST_EMAIL` user but explicitly NOT inherit any storage state.

The compromise: the new spec uses `test.use({ storageState: { cookies: [], origins: [] } })` to opt OUT of the chromium project's default `storageState`, and asserts the unauthenticated redirect. This is the smallest delta that gives meaningful HARDEN-01 coverage without provisioning a learner-only account on prod.

Note that this differs slightly from the original CONTEXT.md description of HARDEN-01 ("Sign in as a non-admin learner, navigate..."). The compromise gives 80% of the value (proves the guard fires for non-admin sessions) without requiring a learner-only account on prod. **This decision is one Jarrad explicitly needs to lock — see Decisions That Need Jarrad's Lock below.**
