## BMH Institute test conventions (derived from `src/**/*.test.ts`)

These are the conventions every existing BMH Institute test follows. New `*.test.tsx` files and any new e2e specs should match this house style.

### File header

Every test file opens with a one-line domain comment that names the ticket or guard being tested, then a brief rationale. Examples:

- `src/app/(dashboard)/admin/users/[userId]/edit/actions.test.ts` line 1: `// HARDEN-03: regression for deleteUser. Six branches: self, last-owner, owner-with-peers, non-owner, admin-client-fail, auth-delete-fail.`
- `src/app/(dashboard)/admin/reports/page.test.ts` line 1: `// HARDEN-01: regression that the page calls requireAdmin() before any data fetch. First codebase use of vi.mock. requireAdmin and createClient have side effects (redirect, network) that prevent direct invocation.`
- `src/app/(dashboard)/admin/users/page.test.ts` line 1: `// WR-05: regression that AdminUsersPage calls requireAdmin() before any data fetch. Same shape as reports/page.test.ts (HARDEN-01).`

### Vitest imports

Single combined import block, sorted alphabetically:

```ts
import { afterEach, beforeEach, describe, expect, it, vi } from "vitest";
```

Pure-function tests drop `beforeEach`, `afterEach`, `vi`:

```ts
import { describe, expect, it } from "vitest";
```

### Mock patterns

- `vi.mock(...)` calls live at top of file, before the SUT import. The SUT is imported AFTER all `vi.mock` calls. See `src/app/(dashboard)/admin/users/[userId]/edit/actions.test.ts` lines 17-59.
- Module mocks return inline factories that close over module-scope mutable state. Example: `let actor: Profile = ...; vi.mock("@/lib/auth/guard", () => ({ requireAdmin: vi.fn(async () => actor) }));`
- Supabase client mocks return a chainable stub object with `select`, `eq`, `in`, `order`, `limit`, `maybeSingle`, plus a `.then` shim so `await chain` works:
  ```ts
  const chain = {
    select: () => chain,
    eq: () => chain,
    in: () => chain,
    order: () => chain,
    limit: () => chain,
    maybeSingle: async () => ({ data: null, error: null }),
    then: (r: (v: { data: unknown[]; error: null }) => unknown) =>
      Promise.resolve({ data: [], error: null }).then(r),
  };
  ```
- `vi.hoisted` is used in Sandra CRM but NOT in any current BMH Institute test. The new RTL smoke file may introduce it; if it does, document the import (`vi.hoisted` from `vitest`).

### `describe` and `it` naming

- `describe` block names match the SUT export, often with the ticket suffix: `describe("deleteUser (HARDEN-03)", ...)`, `describe("AdminReportsPage", ...)`, `describe("AdminUsersPage (WR-05)", ...)`.
- `it` blocks are full sentences in present tense, no leading "should": `it("refuses to delete the acting admin themselves", ...)`, `it("calls requireAdmin before creating a Supabase client", ...)`.

### `beforeEach` and `afterEach`

- `beforeEach` resets module-scope mock state and re-installs default mock implementations.
- `afterEach` always calls `vi.clearAllMocks()`.

### Redirect assertions

Next.js `redirect()` throws an internal error. The convention is a small helper:

```ts
function makeRedirectError(path: string) {
  const e = new Error("NEXT_REDIRECT");
  (e as Error & { digest: string }).digest = `NEXT_REDIRECT;replace;${path};307;`;
  return e;
}
```

Then assert on the digest:

```ts
await expect(AdminReportsPage()).rejects.toMatchObject({
  digest: expect.stringContaining("NEXT_REDIRECT;replace;/login;307;"),
});
```

See `src/app/(dashboard)/admin/reports/page.test.ts` lines 38-42 and 65-72.

### Integration test conventions

- Throwaway-user pattern: every test creates an `auth.users` row with a randomised email like `harden-04-${randomBytes(8).toString("hex")}@bmh.invalid`, runs the assertion, then `try/finally` deletes the user. No truncation of seed data, no shared fixtures.
- Service-role admin client built at module scope with a guard that the env vars exist; otherwise tests skip via the `describe.skip` (current `actions.integration.test.ts`) or import-time throw (current `answer-options-isolation.integration.test.ts`).
- `{ timeout: 30_000 }` per-test timeout for Supabase calls.

### Co-location

Test files always sit next to the file under test, same directory, same basename, `.test.ts` / `.test.tsx` / `.integration.test.ts` suffix. There is no `tests/` directory under `src/` and no centralised mocks folder.

Concrete excerpt to copy (CRM lines 1-33):

```ts
import "@testing-library/jest-dom/vitest";
import { cleanup } from "@testing-library/react";
import { afterEach, vi } from "vitest";

// jsdom doesn't fully wire window.localStorage. Install a minimal in-memory
// Storage shim so components that read/write localStorage on mount can be
// rendered without crashing. Cleared between tests via afterEach.
const store = new Map<string, string>();
const ls: Storage = {
  get length() {
    return store.size;
  },
  clear: () => store.clear(),
  getItem: (key) => (store.has(key) ? (store.get(key) ?? null) : null),
  key: (index) => Array.from(store.keys())[index] ?? null,
  removeItem: (key) => {
    store.delete(key);
  },
  setItem: (key, value) => {
    store.set(key, String(value));
  },
};
Object.defineProperty(window, "localStorage", {
  configurable: true,
  get: () => ls,
});

afterEach(() => {
  cleanup();
  store.clear();
  vi.clearAllMocks();
});
```
