## Phase Boundary

Bring BMH Institute's testing setup to parity with the Sandra CRM testing standard, and replace the five HUMAN-UAT items left open at the close of Phase 01 with automated specs.

Sandra CRM ships four test surfaces: Node-env unit (`*.test.ts`), jsdom RTL (`*.test.tsx`), Postgres-backed integration (`*.integration.test.ts`), and full-stack Playwright e2e. BMH Institute today ships only the first two of those four (unit + integration). The two that exist are working; this phase fills the two that are missing (RTL + e2e) and wires the e2e suite to actually exercise the security boundaries shipped in Phase 01.

In scope:

- Update `package.json` scripts so a single command runs the new suites and the existing `verify` covers the RTL layer.

Out of scope:

- Migrating existing `*.test.ts` unit files to RTL.
- Adding test coverage for surfaces outside Phase 01's HUMAN-UAT list (those land in Phase 04 — TYPE-01, TEST-01..03).

## Implementation Decisions

### Test architecture (locked, mirroring Sandra CRM)
- Keep four parallel suites: unit (Node), RTL (jsdom), integration (Postgres), e2e (Playwright).
- RTL config sits at `vitest.rtl.config.ts`, includes `src/**/*.test.tsx`, environment `jsdom`, setup file `vitest.rtl.setup.ts`.
- RTL setup file imports `@testing-library/jest-dom/vitest`, installs an in-memory `localStorage` shim, runs `cleanup()` and `vi.clearAllMocks()` in `afterEach` (verbatim Sandra CRM pattern).
- Co-locate `*.test.tsx` files alongside the components they test (matches existing `*.test.ts` convention).

### Playwright layout (locked, mirroring Sandra CRM)
- `e2e/` testDir holds local-dev specs; `e2e-prod/` holds prod read-only smoke specs.
- One-time auth via `e2e/auth.setup.ts` that signs in through the real UI and saves storage state to `e2e/.auth/user.json`. Subsequent specs reuse `storageState`.
- Service-role admin client lives in `e2e/fixtures.ts`. It refuses to run if the URL points at a non-test environment (port the `if (url.includes(prodRef)) throw` guard from Sandra CRM, adapted for our project ref `dhvfsyteqsxagokoerrx`).
- `playwright.config.ts` is already authored and points at `e2e/`. Do not rewrite it — only add the files it expects.
- `playwright.prod.config.ts` is already authored and points at `e2e-prod/`. Do not rewrite it — only add the files it expects.

### Dependencies (locked)
- Add as devDependencies: `@testing-library/react`, `@testing-library/jest-dom`, `@testing-library/user-event`, `jsdom`. Pin to versions compatible with React 19.
- Do not pull in `dotenv` — match the existing minimal `.env.test.local` parser pattern from `vitest.integration.config.ts`.

### Scripts (locked)
- Add `"test:rtl": "vitest run --config vitest.rtl.config.ts"`.
- Update `"verify": "npm run typecheck && npm run test && npm run test:rtl"` (mirrors Sandra CRM exactly — verify still excludes integration + e2e because they hit network and cost time).
- `test:e2e` and `test:prod` already exist; do not add new e2e scripts.

### Replacing Phase 01 HUMAN-UAT items (locked direction, write-path strategy is open)
The five items in `.planning/phases/01-auth-and-access-hardening/01-HUMAN-UAT.md` get replaced one for one:

1. HARDEN-01 admin route guard — Playwright spec, **read-only**, can run against prod. Sign in as a non-admin learner, navigate to each of the four `/admin/reports/*` routes, assert each navigation lands on `/dashboard` with no admin row visible.
2. HARDEN-02 expired invite teardown — Playwright spec, **write-path**. Generates an invite, expires it via service-role UPDATE, walks the redirect, asserts `/login?error=invite_expired`, asserts the `auth.users` row is gone. Needs a test environment.
3. HARDEN-03 deleted-user re-authentication — currently covered by `actions.integration.test.ts` with `describe.skip`. Replace by removing the skip and ensuring `TEST_SUPABASE_*` env vars are populated. Optionally also add a Playwright spec that exercises the same path through the UI.
4. HARDEN-04 cross-course answer options — `answer-options-isolation.integration.test.ts` already exists and asserts the contract. Currently throws at import when env vars are absent. Replace by ensuring env vars are populated and the file runs cleanly.
5. Populate `TEST_SUPABASE_*` env vars and run integration suite — this is the meta-step. The phase produces a documented setup script (or README block) that captures: which env vars are required, where they come from, and what command sequence verifies the suite is wired correctly. The actual secret population stays manual.

### Test-data strategy for write-path e2e (Claude's Discretion — flag for review)
Project memory `[No test Supabase project]` says: keep e2e against prod, read-only; don't propose a separate `bmh-institute-test` project. Items 1, 4 (and 5 modulo env vars) honor that constraint as-is. Items 2 and 3 do not — they require write operations against `auth.users` and the `invites` table that should not run on prod.

Three viable paths; planner should default to the cheapest one and surface the trade for Jarrad's lock:

- **Path A (default)** — keep destructive items manual. Replace items 1 and 4 with Playwright specs against `playwright.prod.config.ts` (read-only). For items 2, 3, 5: tighten the existing integration tests (remove `describe.skip` on `actions.integration.test.ts`; remove the import-time throw in `answer-options-isolation.integration.test.ts`) and document the env-var setup so the next contributor can run them locally. Items 2 and 3 stay as HUMAN-UAT entries until a test environment exists.
- **Path B** — use Supabase **ephemeral branches** (Supabase MCP `create_branch` / `delete_branch`) per CI run. Each branch is a copy of the prod schema with isolated data. Spec lifecycle: create branch in `globalSetup`, point `TEST_SUPABASE_*` at it, run, delete in `globalTeardown`. Honors the "no permanent test project" rule.
- **Path C** — service-role on prod with strict prefix-based cleanup (e.g. `e2e-test-{run-id}@bmhgroup.com` users, deleted in afterEach). Highest blast radius; smallest infra overhead.

Recommendation: Path A as the default for this phase, with Path B captured as a follow-up phase if Jarrad wants the destructive items automated later. **Confirm with Jarrad before planning the destructive-item specs.**

<specifics>
## Specific Ideas

- The RTL `localStorage` shim in `vitest.rtl.setup.ts` is non-obvious; Sandra CRM hit jsdom gaps during its rollout. Copy the shim verbatim.
- `playwright.config.ts` already specifies `e2e/.auth/user.json` as the storage-state path. The `auth.setup.ts` spec must write to exactly that path or every other spec will fail with "no storage state".
- The fixtures' "refuse to run against dev/prod" guard should reference the BMH Institute prod project ref (`dhvfsyteqsxagokoerrx`), not Sandra CRM's (`copflsklaefwzipsrjqz`). Easy to copy-paste-miss.
- `webServer.command` in `playwright.config.ts` is `npx next dev -p 3200` (port 3200, not 3100). Specs that hardcode `localhost:3100` will not work; use `baseURL` from the config or relative paths.
- Integration tests already pass when env vars exist (per Phase 01 verification). The blocker is purely the env-var population, not test correctness.
- `answer-options-isolation.integration.test.ts` throws at module level if env vars are missing. That should change to `describe.skipIf(!envPresent)` so the suite reports "skipped" instead of erroring out — unblocks `npm run test:integration` for contributors who haven't set up env vars yet.

</specifics>
